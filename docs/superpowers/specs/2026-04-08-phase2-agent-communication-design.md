# Phase 2: Agent Communication -- Design Specification

Created: 2026-04-08 06:11 UTC

## Overview

Add peer-to-peer communication between Claude Code agents and shared context, built entirely on the filesystem with a FileWatcher for real-time notifications. No in-memory message bus, no knowledge viewer UI, no message log panel -- the filesystem is the source of truth and the debug UI.

## Scope

- Workspace filesystem structure (`.caam/`) for agent state and shared knowledge
- Agent presets persisted in SQLite for cross-project reuse
- Agent bootstrap: spawn modal with purpose editor and preset quicklist
- FileWatcher (chokidar) on `agents/*/inbox.md` with direct terminal injection
- Auto-maintained `agents.md` listing active agents

### Explicitly out of scope

- Knowledge viewer panel in frontend
- Inter-terminal message log UI
- In-memory message bus (deferred to Phase 3 if needed)
- Workflow orchestration

---

## Section 1: Workspace Filesystem Structure

When a workspace is activated, the system creates (if missing) a `.caam/` directory inside the project root:

```
MyProject/
  .caam/
    shared/
      context.md           # Shared project context, written by user or agents
      agents.md            # Auto-generated: list of active agents (see Section 4)
    agents/
      code-reviewer/
        purpose.md          # Agent role and instructions
        inbox.md            # Messages from other agents (append-only)
      architect/
        purpose.md
        inbox.md
```

### Design decisions

- **`.caam/` inside the project** -- each project has its own agents and context, not global.
- **No `outbox.md`** -- redundant with the recipient's `inbox.md`.
- **`inbox.md` is append-only** -- each message is appended with a header:
  ```markdown
  ## From: architect @ 2026-04-08T10:30:00Z
  Please review the API module for error handling gaps.
  ```
- **`shared/` is free-form** -- agents read/write any files here. The system does not watch `shared/` for automated actions (agents access it on their own initiative).

## Section 2: Agent Presets (SQLite)

Presets are stored in the existing SQLite database (`~/.caam/caam.db`) for cross-project reuse.

### Schema

```sql
CREATE TABLE agent_presets (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  purpose TEXT NOT NULL,
  use_count INTEGER DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

### API endpoints

- `GET /api/agent-presets` -- list all presets, ordered by `use_count` descending
- `POST /api/agent-presets` -- create a new preset `{ name, purpose }`
- `PUT /api/agent-presets/:id` -- update name/purpose
- `DELETE /api/agent-presets/:id` -- delete a preset

When a preset is used to spawn an agent, `use_count` is incremented.

## Section 3: Agent Bootstrap -- Spawn Flow

### New spawn flow

1. User clicks "Spawn" (sidebar) or "+" (empty slot in grid)
2. **SpawnModal** opens (blocking modal):
   - **Name** field (required)
   - **Purpose editor** -- large monospace textarea for writing/editing the purpose.md content
   - **Preset quicklist** -- list of saved presets on the left side, ordered by use_count. Click populates name + editor. User can modify before confirming.
   - **"Save as preset"** button -- saves current name+purpose as a new preset (or updates if name matches)
   - **"Spawn"** / **"Cancel"** buttons
3. On Spawn click:
   - Server creates `.caam/agents/<name>/` directory with `purpose.md` containing the editor content
   - Creates empty `inbox.md`
   - Spawns terminal with `claude --dangerously-skip-permissions` in workspace cwd
   - Injects into terminal: `Read your purpose and instructions at .caam/agents/<name>/purpose.md and follow them.`
   - Increments `use_count` if a preset was used
   - Regenerates `shared/agents.md` (see Section 4)

### Purpose template

The purpose.md includes a standard communication section appended by the system:

```markdown
<user-written purpose content>

---

## Communication

To see which agents are available, read:
  .caam/shared/agents.md

To send a message to another agent, append to:
  .caam/agents/<target-name>/inbox.md

Use this format:
  ## From: <your-name> @ <timestamp>
  <message body>

Your inbox is at .caam/agents/<your-name>/inbox.md
Check it when notified.
```

### WebSocket protocol addition

The spawn message gains optional fields:

```typescript
{ type: 'terminal:spawn', name: string, cwd: string, purpose?: string, presetId?: number }
```

The server handles purpose file creation and preset use_count increment internally.

## Section 4: Active Agents Registry (`agents.md`)

The file `.caam/shared/agents.md` is auto-generated by the server whenever an agent is spawned or terminated. It is **fully regenerated** each time (not appended) to avoid stale state.

### Format

```markdown
# Active Agents

## code-reviewer
- **Status**: running
- **Purpose**: Review code for bugs and style issues

## architect
- **Status**: running
- **Purpose**: Design system architecture
```

Each entry includes the agent name, status, and the first line of its purpose.md as summary.

### Triggers for regeneration

- Agent spawned (terminal:created)
- Agent terminated (terminal:exited, Close Pane, manual `exit`)
- On spawn, the terminated agent is removed from the list

## Section 5: FileWatcher and Direct Injection

### KnowledgeWatcher module

A new module at `packages/server/src/modules/knowledge/` using `chokidar` to watch the `.caam/` directory of the active workspace.

**Watches:** `agents/*/inbox.md` for changes (new messages).

**Does not watch:** `shared/` -- no automated action on shared files.

### Lifecycle

- Starts when a workspace is activated (first terminal spawn sets the cwd)
- Stops and restarts when the workspace changes
- Stops when the server shuts down

### Injection flow

When chokidar detects a change to `agents/<name>/inbox.md`:

1. Debounce: 300ms to batch rapid writes
2. Look up terminal by agent name in TerminalRegistry
3. If terminal exists and is running, inject:
   ```
   [CAAM] New message in your inbox. Read .caam/agents/<name>/inbox.md
   ```
4. Injection uses `TerminalSession.write()` -- same mechanism as user keyboard input

### Edge cases

- **Agent not running** -- watcher ignores inbox changes for agents without an active terminal
- **Rapid writes** -- chokidar debounce at 300ms; one injection per batch
- **Workspace switch** -- old watcher is closed, new one starts on the new directory
- **`.caam/` doesn't exist yet** -- watcher creates the directory structure on first spawn, then starts watching

## Section 6: Frontend Changes

### SpawnModal (new component)

Opened by:
- "Spawn" button in sidebar (existing)
- Click on EmptySlot in grid (existing)

Both paths open the same modal. On successful spawn, the modal closes and the terminal appears in the grid (existing auto-assignment logic).

### Sidebar

No structural changes. The kill button in the sidebar continues to work as today (kills process, collapses pane). The server regenerates `agents.md` on terminal exit.

### No new panels

No knowledge viewer. No message log. The filesystem is the debug UI.

## Architecture Summary

```
                     SpawnModal
                         |
                   [name + purpose]
                         |
                         v
  +--------------------------------------------------+
  |  Server (app.ts)                                  |
  |                                                   |
  |  TerminalRegistry ←→ TerminalSession (node-pty)   |
  |       ↑                      ↑                    |
  |       |                      | write()            |
  |  KnowledgeWatcher            |                    |
  |  (chokidar on .caam/)        |                    |
  |       |                      |                    |
  |  inbox.md changed ──────→ inject notification     |
  |                                                   |
  |  AgentBootstrap                                   |
  |  - creates .caam/agents/<name>/                   |
  |  - writes purpose.md                              |
  |  - regenerates shared/agents.md                   |
  |                                                   |
  |  PresetRepository (SQLite)                        |
  |  - CRUD agent_presets table                       |
  +--------------------------------------------------+
```

All communication between agents flows through the filesystem. The server's role is limited to: creating agent directories, watching for inbox changes, injecting notifications, and maintaining the agents.md registry.
